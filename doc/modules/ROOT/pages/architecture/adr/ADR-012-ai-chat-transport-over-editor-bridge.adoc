= ADR-012 — BlockNote AI Traffic Routes Through EditorBridge, Not Direct Fetch

== Status

Accepted

== Date

2026-07-16

---

== Context

`@blocknote/xl-ai`'s `AIExtension` is designed, out of the box, to call `fetch`/streaming
requests directly from browser JS to a configured API URL (via `DefaultChatTransport` from
the Vercel `ai` SDK), with the API key held server-side.

This conflicts with the standing rule for this project (`CLAUDE.md`, `src/bridge/`):
the in-browser BlockNote editor must never get raw or unrestricted network access —
`EditorBridge`/`QWebChannel` is the single declared trust boundary between JS and the
desktop app. `AIExtension`'s default transport would give the `QWebEngineView` content a
direct network egress path, bypassing that boundary for the first time.

An allowlist exception (permit exactly one URL through `QWebEngineView`'s network layer)
was considered and rejected: it would be the first crack in an otherwise absolute rule,
and the rule was called out explicitly in the code-analysis pass ahead of this work
(see ADR-010) as the property to protect while iterating quickly on AI features.

---

== Decision

BlockNote AI features use a custom `ChatTransport` implementation (matching the `ai` SDK's
`ChatTransport` interface consumed by `AIExtension`) that does not call `fetch` at all.
Instead, it forwards each request through `EditorBridge`/`QWebChannel` to C++, which makes
the actual HTTP request to `cppwiki_server` and streams the response back to JS via bridge
signals.

`QWebEngineView` retains zero direct network access; the only network egress for AI, as
for everything else, is mediated by `EditorBridge`.

---

== Consequences

=== Positive

* No exception to the "editor gets no raw network access" rule — the trust boundary stays
  singular and absolute rather than acquiring a first carve-out.
* AI requests naturally inherit whatever auth/session context `EditorBridge` already
  attaches to other calls, instead of needing their own credential-passing story into the
  browser.

=== Negative

* Requires writing and maintaining a custom `ChatTransport` against the `ai` SDK's
  streaming contract, instead of using `DefaultChatTransport` as documented upstream —
  more work than the out-of-the-box path, and needs to be kept in sync if BlockNote changes
  that interface in future releases.
* Streaming responses must be relayed as a sequence of bridge signals (chunk-by-chunk)
  rather than a native browser `ReadableStream`, which is more plumbing on both the C++ and
  JS sides than the default integration.

== Addendum (2026-07-16) — Serverless Builds: Local-Key Fallback

=== Context

CppWiki is offline-first, and `cppwiki_server` is not mandatory: a user can run the desktop
app against local storage only, with no backend configured. The original decision above
assumed the AI provider API key lives server-side and every AI request is proxied through
`cppwiki_server`. That leaves no path to AI features at all for a serverless install.

=== Decision

`EditorBridge`'s custom `ChatTransport` gets two backends behind the same interface:

* *Server-mediated* (default, unchanged): if a backend is configured, the request is
  forwarded to `cppwiki_server`, which holds the API key and calls the AI provider.
* *Local-key fallback*: if no backend is configured, the transport instead reads a
  user-supplied API key from local secure storage (`qt-keychain`, the same mechanism
  `AuthTokenStore` already uses — see the admin CLI's planned reuse of it in
  `Admin_Surface_Backlog.adoc`) and calls the AI provider directly from C++.

In both cases, the call still originates from C++ inside `EditorBridge`, not from JS.
`QWebEngineView` gains no new network egress path either way — the fallback only changes
*where* the key and the outbound HTTP call live (local keychain + direct provider call vs.
server-held key + proxied call), not *who* makes the call. This keeps the transport a single
interface with two implementations selected by whether a backend is configured, rather than
a special case inside the editor/JS layer.

The settings UI's AI section (ADR-013-adjacent settings work, `SegmentedControl` AI page)
must expose the local API key field only when no backend is configured, and must state
plainly that in this mode requests go directly from the user's machine to the AI provider,
not through `cppwiki_server`.

=== Consequences

==== Positive

* Serverless/offline-first users keep access to AI features instead of losing them
  outright — this was a real gap in the original decision.
* No exception to the "editor gets no raw network access" rule: both backends are reached
  from the same C++-side transport, never from JS.

==== Negative

* Two code paths (server-mediated, local-key) to implement, secure, and test instead of
  one; the local-key path needs its own error handling for provider-side rate limits/costs
  that `cppwiki_server` would otherwise absorb or manage centrally.
* Local-key users bear their own AI provider costs and rate limits directly, and are
  trusting their own machine's keychain rather than server-side secret management — this
  must be communicated clearly in the settings UI, not left implicit.
* The two backends must be kept behavior-compatible (same `ChatTransport` contract) as
  BlockNote's AI extension evolves, doubling the maintenance surface flagged as a risk in
  the original decision above.

== Addendum (2026-07-19) — Inline Ghost-Text Suggestions and Structured Tool-Call Output

=== Context

Two more AI-transport-relevant features landed after the original decision and its
local-key addendum above, both surfaced through direct use of the running app rather than
being anticipated up front.

*Inline ghost-text suggestions* (issue #59): users wanted a continuous, Copilot/Obsidian-style
inline completion — text predicted and shown as dimmed "ghost text" right at the cursor while
typing, accepted with Tab — rather than the existing manually-triggered toolbar/slash-menu AI
actions. This is a fundamentally different interaction pattern (fired automatically and
repeatedly on typing pauses, not once per explicit user action).

*Structured tool-call output* (issue #65): `@blocknote/xl-ai`'s `AIExtension` (used by the
rewrite toolbar button and the AI slash-menu items) turned out not to render plain-text
responses at all. Its own `defaultAIRequestSender` (`node_modules/@blocknote/xl-ai`) puts a
tool name and JSON Schema in the request body
(`toolDefinitions.applyDocumentOperations`, built from the current document), and its
`setupToolCallStreaming` only applies document changes from *tool-call* message parts
(`tool-input-start`/`-delta`/`-available` `UIMessageChunk`s) — it silently ignores plain
`text-delta` parts. `BridgeChatTransport` was only ever emitting `text-delta` chunks (matching
this ADR's original "plain text streaming, no tool-calling" MVP scope), so responses arrived
successfully end to end (visible in `cppwiki_server`'s logs) but never surfaced in the editor.

=== Decision

==== Inline suggestions bypass `@blocknote/xl-ai` entirely

Rather than force the continuous, automatic inline-completion pattern through
`AIExtension`/`AIMenuController` (designed around discrete, user-triggered invocations), inline
suggestions are a separate Tiptap extension
(`frontend/editor/src/extensions/inlineSuggestionExtension.ts`, wired into
`useCreateBlockNote()` alongside `createAIExtension()`), backed by its own debounce/cancellation
state machine (`inlineSuggestionScheduler.ts`, unit-tested independent of ProseMirror) and its
own bridge call (`frontend/editor/src/bridge/inlineCompletion.ts`, `startAiRequest(prompt,
"", "inline")`).

It still goes through the same `EditorBridge`/`QWebChannel` trust boundary as every other AI
request (this ADR's core decision is unchanged), but consumes the response as a single
concatenated string, not a `ReadableStream<UIMessageChunk>` — there is no `ai` SDK `Chat`
session involved for this feature, since there is no BlockNote AI menu UI to drive.

Key behavior, since none of this is discoverable from the code alone without reading issue #59:

* *Debounce*: 450ms after the last relevant transaction (`inlineSuggestionDebounceMs`,
  `frontend/editor/src/constants.ts`) before a completion request fires — chosen from the
  issue's suggested 300-600ms range: long enough to avoid firing on every keystroke, short
  enough that a pause reads as "waiting for a suggestion" rather than a stall.
* *Context*: up to 800 characters of plain text immediately preceding the cursor
  (`inlineSuggestionContextChars`) — local context only, not the whole document, to keep
  prompts small given the self-hosted provider's higher per-request latency.
* *Trigger condition*: only when the selection is collapsed (an empty cursor, not a text
  range) and the document actually changed or the selection moved; scheduling is skipped
  (and any pending request cancelled) while `editor.replaceBlocks()` is programmatically
  swapping the whole document (opening a different page, applying a conflict resolution —
  see `isReplacingDocument` in `inlineSuggestionExtension.ts`), not just on local typing.
* *Cancellation*: any further doc-changing transaction (continued typing) cancels a pending
  debounce timer or in-flight request and dismisses an already-showing suggestion. Cancellation
  is client-side only — once `startAiRequest` has handed the request to C++, the outbound HTTP
  call to the provider is not aborted server-side; a cancelled request's eventual result is
  simply ignored.
* *Display*: rendered as a ProseMirror widget decoration (`.cppwiki-ghost-text`, dimmed,
  `aria-hidden`) positioned at the cursor, not inserted into the document.
* *Accept*: Tab inserts the suggestion text at the cursor and clears the decoration. The
  keyboard shortcut's `priority` is set higher than Tiptap/BlockNote's implicit default so it
  gets first refusal on Tab, but returns `false` (falls through to BlockNote's own Tab
  handling, e.g. list indent) whenever there is no suggestion currently showing.
* *Opt-in*: gated by its own setting, `ai/inlineSuggestionsEnabled`
  (`ProgramSettings::AiInlineSuggestionsEnabled()`), deliberately independent of the existing
  `ai/featuresEnabled` (rewrite/slash-menu) toggle — a user can want continuous inline
  predictions without the toolbar/slash-menu AI actions, or vice versa.
* *Prompting*: the server selects a short, single-continuation system prompt for
  `mode: "inline"` requests (`constants::kAiSystemPromptInline`,
  `src/core/constants.h`) — deliberately different from the rewrite/autocomplete prompts, to
  keep latency and output size down given how often this fires.

==== Rewrite/autocomplete switch to structured tool-call output

`BridgeChatTransport` (`frontend/editor/src/bridge/aiChatTransport.ts`) now reads the tool
name and JSON Schema xl-ai puts in `options.body.toolDefinitions.applyDocumentOperations` and
forwards both through the bridge alongside the existing prompt/context/mode
(`EditorBridge.startAiRequest`'s new optional `toolName`/`toolSchemaJson` parameters). The
plain-text path (`text-delta` chunks) is kept as a fallback for callers that don't supply a
tool schema (i.e. the inline-suggestions path above, and any future non-xl-ai caller).

On the C++ side, `QEditorBridge::CallProviderAndRelay` threads the schema through to whichever
backend handled the request (server-mediated or local-key) and, when the provider replies with
a structured tool call instead of prose, relays it via a dedicated `aiToolCallReceived(request_id,
tool_name, arguments_json)` signal rather than chunked text. `cppwiki_server`'s
`AiChatService::BuildRequestBody` builds an OpenAI-compatible `tools`/`tool_choice` request
(forcing the specific function call) when the incoming `AiChatRequestDto` carries a tool
name/schema, and extracts `choices[0].message.tool_calls[0].function.arguments` instead of
`choices[0].message.content` for the response.

Back on the JS side, `BridgeChatTransport` re-emits the parsed arguments as a
`tool-input-start`/`tool-input-delta`/`tool-input-available` `UIMessageChunk` sequence (rather
than `text-delta`), which is what BlockNote's `setupToolCallStreaming` actually watches for —
closing the gap that caused responses to arrive successfully but never surface in the editor.

=== Consequences

==== Positive

* Inline suggestions get a UX (continuous, low-friction, Tab-to-accept) that would have been
  awkward to force through `AIExtension`'s discrete-invocation menu model, without
  compromising that model for the toolbar/slash-menu features that do fit it well.
* Both new features keep the ADR-012 trust boundary intact: `QWebEngineView` still never gets
  direct network access, regardless of which of the three request "modes" (rewrite,
  autocomplete, inline) or which chunk shape (text vs. tool-call) is in play.
* The rewrite/autocomplete fix makes `AIMenuController`'s built-in diff/preview UI actually
  work, instead of requiring a hand-rolled replacement UI that would have diverged from
  `@blocknote/xl-ai`'s own (and future) UX for that feature.

==== Negative

* The bridge and transport interfaces have grown a third "mode" and a parallel tool-call
  signal/chunk path alongside the original plain-text one — two response shapes
  (`aiChunkReceived`/`text-delta` vs. `aiToolCallReceived`/`tool-input-*`) to keep in sync as
  `@blocknote/xl-ai` or the `ai` SDK's `UIMessageChunk` contract evolves.
* Inline suggestions add a request every ~450ms of typing pause, not just on explicit user
  action — meaningfully higher AI provider request volume/cost than the original
  manually-triggered scope this ADR was written for.
* The tool-calling path assumes an OpenAI-compatible `tools`/`tool_choice` API shape; a
  self-hosted or alternative provider that doesn't support function-calling in that exact
  shape would need either a translation layer or would only work in plain-text mode (rewrite/
  autocomplete would then silently fail to surface responses again, same as before this fix).
