= Server Code Explained

This document explains the current `src/server/` code in plain language.

Its goal is not to describe every line.
Its goal is to answer:

* what each server layer is responsible for;
* how a request moves through the code;
* why some helper files exist;
* where to make changes when you want to extend behavior.

'''

== 1. High-Level Flow

Server startup is intentionally linear:

. `main.cc` parses CLI args into `RuntimeConfig`.
. `RunServer(...)` configures logging, generates userver static config YAML, and starts userver.
. `RegisterCppWikiComponents(...)` registers handlers and the auth checker factory.
. userver loads the generated static config and instantiates the registered components.

At request time the flow is:

. userver routes the request to a handler by name/path from the generated static config;
. the handler prepares response headers and tracing/logging tags;
. the handler parses request JSON if needed;
. the handler calls a small in-memory service;
. the handler wraps the result into a response DTO;
. the DTO is serialized to JSON and returned to userver.

'''

== 2. Directory Map

`src/server/` is split by responsibility:

* `app/`
  Startup entrypoint and top-level run function.
* `config/`
  Runtime config parsing and generated userver static config.
* `components/`
  userver component registration.
* `handlers/`
  HTTP endpoints.
* `dto/`
  Request/response transport models and JSON adapters.
* `service/`
  In-memory business logic stubs for locks and presence.
* `middleware/`
  Cross-cutting request behavior such as auth and tracing tags.
* `assets/`
  Embedded OpenAPI JSON and Swagger UI HTML.

'''

== 3. Startup Layer

=== `src/server/app/main.cc`

Responsibility:

* parse CLI args;
* convert them into `RuntimeConfig`;
* call `RunServer(config)`;
* log a top-level fatal error if startup fails.

This file should stay tiny.

=== `src/server/app/server_application.{h,cc}`

Responsibility:

* provide `RunServer(const RuntimeConfig&)`;
* configure logging;
* ask `RuntimeConfig` for generated static config YAML;
* write that YAML into a temporary file;
* create a minimal userver component list;
* register project components;
* call `userver::components::Run(...)`.

This is the bootstrap boundary between "our config" and "userver runtime".

'''

== 4. Config Layer

=== `src/server/config/runtime_config.{h,cc}`

Responsibility:

* define the small runtime contract:
** `config_path`
** `bind_host`
** `port`
** `log_level`
** `swagger`
* parse CLI options with `CLI11`;
* load `config/server.yaml` through `reflect-cpp`;
* apply CLI overrides on top of file config;
* expose final values via `Host()`, `Port()`, `LogLevel()`;
* delegate static config generation to `MakeStaticConfigYaml(...)`.

Important design point:

NOTE: `RuntimeConfig` is *not* the full userver config. It is only the small human-facing config contract for this project.

=== `src/server/config/static_config_factory.{h,cc}`

Responsibility:

* convert the small `RuntimeConfig` values into the large userver `components_manager` YAML;
* define the internal C++ DTO tree needed to emit that YAML through `reflect-cpp`;
* register handler config blocks such as:
** `handler-health`
** `handler-openapi`
** `handler-locks`
** `handler-presence`
** `handler-protected-page`
** `handler-swagger-ui`

Why this file exists:

userver wants a fairly verbose static config.
We do not want that verbosity leaking into `RuntimeConfig`.

So this file acts as a translation layer:

* input: simple runtime config;
* output: full userver startup config.

'''

== 5. Component Registration

=== `src/server/components/cppwiki_server_component.{h,cc}`

Responsibility:

* append all handler classes to `userver::components::ComponentList`;
* register the auth checker factory under auth type `cppwiki-auth-checker`.

The auth registration matters because protected routes in generated static config refer to this auth type.

NOTE: If a handler is registered here, it must also exist in generated static config. If an auth type is referenced in generated static config, it must be registered here.

That coupling is the main thing to remember in this file.

'''

== 6. Handler Helpers

=== `src/server/handlers/handler_utils.{h,cc}`

Responsibility:

* apply common HTTP headers;
* attach tracing/request tags;
* prepare common response styles:
** JSON API response
** HTML response
** JSON document response

Why this file exists:

Without it, every handler repeated the same boilerplate:

* get `HttpResponse`
* set CORS/content headers
* call tracing/tagging helper

This file keeps handlers focused on request-specific logic.

'''

== 7. HTTP Handlers

=== `health_handler`

Responsibility:

* `GET /api/v1/health`
* return a small success envelope with service name and `"ok"` status.

This is the simplest example of the handler pattern:

. prepare headers/tags;
. build DTO;
. wrap DTO into response envelope.

=== `options_handler`

Responsibility:

* answer CORS preflight requests;
* return `204 No Content`.

=== `openapi_handler`

Responsibility:

* serve the OpenAPI document as JSON;
* return the embedded content from `assets/openapi_spec.h`.

=== `swagger_ui_handler`

Responsibility:

* serve the Swagger UI HTML page;
* return the embedded content from `assets/swagger_ui_page.h`.

=== `lock_handler`

Responsibility:

* handle lock operations on `/api/v1/locks/\{document_id}`;
* dispatch by HTTP method:
** `GET` = read lock state
** `POST` = acquire
** `PUT` = heartbeat
** `DELETE` = release or force release

What it does internally:

* parses request body into `LockRequestDto` through the shared JSON adapter;
* extracts owner from JSON first, then from header fallback, then defaults to `"anonymous"`;
* calls `LockService`;
* converts service result into a response DTO and success/error envelope.

=== `presence_handler`

Responsibility:

* handle presence operations on `/api/v1/presence/\{workspace_id}`;
* for `POST`, update presence heartbeat;
* for `GET` and after `POST`, return current presence list.

What it does internally:

* parses request JSON into `PresenceHeartbeatRequestDto`;
* defaults missing values to:
** `user_id = "anonymous"`
** `scope = "view"`
* calls `PresenceService`;
* returns a typed presence response DTO.

=== `protected_page_handler`

Responsibility:

* simple protected smoke endpoint;
* verifies the protected-route/auth plumbing works;
* currently returns a stub message.

This handler is mainly an auth boundary test endpoint.

'''

== 8. DTO Layer

=== `src/server/dto/json_adapter.h`

Responsibility:

* shared conversion boundary between:
** `userver::formats::json::Value`
** `reflect-cpp` typed structs

It exposes two generic helpers:

* `ParseJsonBody<T>(...)`
* `ToJsonValue(...)`

This is the file that keeps JSON conversion policy centralized.

=== `src/server/dto/response_envelope.{h,cc}`

Responsibility:

* define common API envelope shapes;
* provide:
** `SuccessEnvelope<T>`
** `ErrorEnvelope`
** `ErrorDto`
* serialize them back into `userver::formats::json::Value`.

The API contract here is:

[source,json]
----
{
  "apiVersion": 1,
  "ok": true,
  "result": { ... }
}
----

or

[source,json]
----
{
  "apiVersion": 1,
  "ok": false,
  "error": {
    "code": "...",
    "message": "..."
  }
}
----

=== `health_response`

Responsibility:

* define the typed health result DTO.

=== `lock_response`

Responsibility:

* define:
** request DTO for lock payloads;
** internal lock action result struct;
** response DTOs for lock state.

=== `presence_response`

Responsibility:

* define:
** request DTO for presence heartbeat payloads;
** response DTOs for returned presence users.

Important note about DTOs:

NOTE: DTOs are transport models. They should describe what goes over HTTP, not necessarily the internal storage shape.

'''

== 9. Service Layer

=== `src/server/service/lock_service.{h,cc}`

Responsibility:

* keep in-memory lock state per `document_id`;
* support:
** acquire
** heartbeat
** release
** force release
** read owner

Current characteristics:

* mutex-protected;
* in-memory only;
* no persistence;
* same-owner acquire is treated as refresh;
* different-owner acquire is rejected.

This is stub business logic, not final production lock management.

=== `src/server/service/presence_service.{h,cc}`

Responsibility:

* keep in-memory presence state grouped by workspace and user;
* support:
** update heartbeat
** list current presence for one workspace

Current storage shape:

* `workspace_id -> user_id -> PresenceInfo`

This is intentionally simple and efficient for the current stub phase.

'''

== 10. Middleware Layer

=== `src/server/middleware/logging_middleware.{h,cc}`

Responsibility:

* attach tracing/span tags from request data;
* propagate `x-request-id` back to response if present;
* tag requests with:
** method + URL
** `document_id` when present
** `workspace_id` when present

This is not business logic.
It is request observability plumbing.

=== `src/server/middleware/auth_checker_impl.{h,cc}`

Responsibility:

* implement the auth checker referenced by protected handlers;
* currently reject protected traffic with a stub response.

This is a placeholder for future real auth integration.

'''

== 11. Asset Layer

=== `src/server/assets/openapi_spec.h`

Responsibility:

* hold the embedded OpenAPI JSON document.

=== `src/server/assets/swagger_ui_page.h`

Responsibility:

* hold the embedded Swagger UI HTML page.

Why these files exist:

The handlers should describe HTTP behavior, not be giant string containers.

'''

== 12. How To Change Things

If you want to change host/port/log level loading:

* edit `runtime_config.{h,cc}`

If you want to change userver startup config shape:

* edit `static_config_factory.cc`

If you want to add a new endpoint:

. add a handler in `handlers/`
. add DTOs if needed in `dto/`
. add a service if the handler needs domain logic
. register the handler in `cppwiki_server_component.cc`
. add its config block in `static_config_factory.cc`

If you want to change shared JSON/request/response glue:

* edit `dto/json_adapter.h`
* edit `dto/response_envelope.{h,cc}`

If you want to change lock/presence behavior:

* edit `service/lock_service.cc`
* edit `service/presence_service.cc`

If you want to change tracing/CORS preparation:

* edit `handlers/handler_utils.cc`
* edit `middleware/logging_middleware.cc`

If you want to change protected-route auth behavior:

* edit `middleware/auth_checker_impl.cc`

'''

== 13. Mental Model

The easiest way to understand this server is:

* `RuntimeConfig` = small human-facing config
* `StaticConfigFactory` = translator into userver startup YAML
* `Component registration` = tell userver which classes exist
* `Handler` = HTTP adapter
* `DTO` = wire format
* `Service` = in-memory logic
* `Middleware/helper` = cross-cutting request plumbing

If you keep those boundaries in mind, the code stops looking like one large server blob and becomes a small set of very narrow layers.
