= Взаимодействие Collaboration Flow с Couchbase Sync

Этот документ фиксирует, как должен быть устроен путь совместной работы в CppWiki, когда в системе одновременно присутствуют:

* desktop shell на Qt;
* локальное offline-first хранилище через Couchbase Lite;
* backend coordination service `cppwiki-server`;
* репликация через Couchbase Sync Gateway;
* аутентификация через Authentik / OIDC.

Документ нужен для того, чтобы не смешивать разные обязанности в одну "sync-систему".

'''

== 1. Главная идея

В архитектуре CppWiki есть два разных слоя:

. слой координации редактирования;
. слой репликации данных.

Они связаны одной пользовательской identity, но решают разные задачи.

=== Coordination layer

За него отвечает `cppwiki-server`.

Он определяет:

* кто сейчас владеет lock на документ;
* кому разрешено редактировать документ прямо сейчас;
* кто присутствует в workspace;
* когда lock считается протухшим;
* когда editor должен перейти в read-only режим.

=== Replication layer

За него отвечают `Couchbase Lite` и `Sync Gateway`.

Они определяют:

* где хранится локальная копия документов;
* как локальные изменения переживают offline;
* как изменения уезжают на сервер;
* как изменения приезжают на другие клиенты;
* к каким данным пользователь вообще имеет доступ.

Иными словами:

* `cppwiki-server` отвечает за *операционное состояние совместной работы*;
* `Couchbase Lite + Sync Gateway` отвечают за *доставку и синхронизацию данных*.

'''

== 2. Роли компонентов

== 2.1. Desktop App

Desktop-приложение:

* показывает UI;
* запускает login flow;
* хранит локальную базу Couchbase Lite;
* открывает editor в `editable` или `read-only` режиме;
* запрашивает lock перед редактированием;
* шлёт heartbeat, пока пользователь редактирует;
* запускает репликацию с Sync Gateway;
* отображает lock owner, presence и sync state.

Desktop shell остаётся главным приложением.
Ни `Sync Gateway`, ни `cppwiki-server` не становятся "вторым editor runtime".

== 2.2. CppWiki Server

`cppwiki-server`:

* валидирует JWT;
* извлекает authenticated principal;
* управляет lock ownership;
* принимает heartbeat по lock и presence;
* отдаёт оперативное collaboration state;
* даёт backend boundary для audit/logging/observability.

Сервер не является primary document store для MVP collaboration path.

== 2.3. Couchbase Lite

`Couchbase Lite`:

* хранит локальную offline копию документов;
* позволяет писать изменения локально без зависимости от сети;
* даёт desktop shell устойчивое поведение при временной недоступности backend;
* реплицируется с `Sync Gateway`.

Для пользователя это основной локальный data plane.

== 2.4. Sync Gateway

`Sync Gateway`:

* принимает репликацию от desktop clients;
* отдаёт удалённые изменения обратно клиентам;
* ограничивает доступ к данным через channel-based auth model;
* связывает authenticated user с разрешённым набором документов/workspaces.

`Sync Gateway` не должен быть источником истины для lock ownership.

'''

== 3. Что именно хранится где

=== В `cppwiki-server`

Хранится краткоживущая coordination state:

* lock owner;
* lock expiry / heartbeat freshness;
* presence state;
* audit trail событий lock takeover / force release;
* derived user identity из JWT для request processing.

=== В Couchbase Lite / Sync Gateway

Хранится document data:

* страницы;
* блоки документа;
* metadata документа;
* snapshots / conflict-related artifacts;
* workspace-level persisted entities.

=== В desktop app

Хранится runtime state UI:

* открыт ли editor;
* editable или read-only mode;
* текущий lock status;
* auth session;
* локальный sync status;
* временные UI flags.

'''

== 4. Общая схема взаимодействия

[source,mermaid]
----
flowchart LR
    User[User]
    Desktop[Desktop App]
    Auth[Authentik]
    Server[cppwiki-server]
    Lite[Couchbase Lite]
    Gateway[Sync Gateway]
    Couchbase[(Couchbase Server)]

    User --> Desktop
    Desktop --> Auth
    Desktop --> Server
    Desktop --> Lite
    Lite <--> Gateway
    Gateway <--> Couchbase
----

Смысл схемы:

* пользователь взаимодействует только с desktop app;
* desktop app делает auth и получает токены;
* desktop app использует токены для `cppwiki-server` и для sync side;
* `cppwiki-server` решает collaboration coordination;
* `Couchbase Lite` и `Sync Gateway` решают document replication.

'''

== 5. Поток открытия документа

Когда пользователь открывает страницу, правильный целевой поток такой:

. Desktop app загружает документ из локального `Couchbase Lite`.
. Desktop app понимает, что документ доступен локально, даже если сеть временно плохая.
. Перед включением полноценного редактирования desktop app обращается к `cppwiki-server` за lock.
. Если lock получен:
** editor открывается в `editable` режиме;
** стартует heartbeat.
. Если lock не получен:
** editor открывается в `read-only` режиме;
** UI показывает текущего owner.

То есть:

* чтение документа может идти из локальной базы;
* право на запись определяется не локальной базой, а lock service.

'''

== 6. Поток редактирования

Во время редактирования система должна работать так:

. Пользователь вносит изменения в editor.
. Desktop shell сериализует обновлённый документ и пишет его в локальный `Couchbase Lite`.
. Параллельно desktop shell поддерживает lock через heartbeat в `cppwiki-server`.
. Если heartbeat перестал проходить или lock потерян:
** desktop shell должен перестать считать документ editable;
** editor должен перейти в read-only;
** UI должен показать причину.

Ключевая мысль:

* локальная запись в `Couchbase Lite` не равна праву редактирования;
* право редактирования подтверждается активным lock.

'''

== 7. Поток репликации

После локальной записи документ синхронизируется отдельно от lock flow.

=== Репликация идёт так

. Изменения уже сохранены локально в `Couchbase Lite`.
. Replicator отправляет их в `Sync Gateway`.
. `Sync Gateway` проверяет channel access для authenticated пользователя.
. Если доступ разрешён, изменения попадают в server-side dataset.
. Другие desktop clients получают их через свою репликацию.

=== Что важно

Это не означает, что другой клиент может начать одновременно писать документ.

Он может:

* получить новые данные;
* отобразить их;
* открыть документ локально;
* но всё равно должен отдельно запросить lock у `cppwiki-server`, если хочет редактировать.

'''

== 8. Почему lock не должен жить внутри sync-документов

Иногда кажется, что lock можно сделать просто отдельным документом в Couchbase Sync. Для MVP это плохая идея.

Причины:

* lock должен быть краткоживущим и heartbeat-driven;
* lock требует быстрых timeout semantics;
* lock требует явного operational ownership;
* lock должен быть связан с request-time auth, а не только с eventual replication;
* репликация по природе eventual, а lock нужен почти как near-realtime coordination state.

Если хранить lock как обычный replicated document:

* появятся гонки на обновление lock record;
* потеряется чёткая семантика "single writer now";
* offline clients начнут видеть устаревший lock state как будто он актуален;
* force release и audit станут гораздо менее надёжными.

Поэтому lock service и replication service должны оставаться разными слоями.

'''

== 9. Как auth связывает оба слоя

Вся система должна опираться на одну и ту же пользовательскую identity.

=== На стороне `cppwiki-server`

JWT нужен для:

* валидации bearer token;
* определения authenticated principal;
* заполнения `lock owner` и `presence user` по умолчанию;
* логирования действий пользователя;
* проверки backend access boundary.

=== На стороне Sync

Та же identity нужна для:

* определения channel membership;
* ограничения доступа к наборам документов;
* репликации только разрешённых данных.

То есть одна и та же user identity должна участвовать в двух местах:

. в coordination layer;
. в replication layer.

Но обязанности этих слоёв всё равно различны.

'''

== 10. Целевой end-to-end сценарий

[source,mermaid]
----
sequenceDiagram
    participant U as User
    participant D as Desktop App
    participant S as cppwiki-server
    participant L as Couchbase Lite
    participant G as Sync Gateway

    U->>D: Open page
    D->>L: Load local document
    L-->>D: Document snapshot
    D->>S: POST /locks/{document_id}
    S-->>D: Lock acquired or denied

    alt Lock acquired
        D->>U: Editor opens editable
        loop While editing
            U->>D: Change document
            D->>L: Save local document
            D->>S: PUT /locks/{document_id} heartbeat
            D->>S: POST /presence/{workspace_id}
            L->>G: Replicate changes
        end
    else Lock denied
        D->>U: Editor opens read-only
    end
----

Этот flow показывает главное:

* editor работает поверх локальной базы;
* право записи даёт lock service;
* репликация идёт параллельно;
* auth identity проходит через оба канала.

'''

== 11. Что должно произойти в ближайшей Phase 7

Следующий этап должен довести архитектуру до минимально жизнеспособного collaboration path.

=== Нужно сделать

* desktop app запрашивает lock перед входом в editable mode;
* desktop app отправляет heartbeat, пока документ открыт для редактирования;
* desktop app переводит editor в read-only, если lock не получен или потерян;
* UI показывает lock owner;
* presence endpoint начинает использоваться как часть рабочего цикла;
* authenticated identity из JWT становится стандартным owner/user id для collaboration endpoints.

=== Пока не нужно делать

* realtime multi-writer collaboration;
* Yjs/CRDT runtime;
* merge UI для одновременного редактирования;
* попытку закодировать lock semantics через replicated documents.

'''

== 12. Вывод

В CppWiki collaboration с Couchbase Sync должен строиться по принципу разделения обязанностей:

* `Couchbase Lite + Sync Gateway` отвечают за document replication;
* `cppwiki-server` отвечает за operational coordination;
* `desktop app` связывает их в единый пользовательский поток;
* `OIDC/JWT` даёт общую identity-модель для обоих слоёв.

Самое важное архитектурное правило:

NOTE: репликация не заменяет lock service, а lock service не заменяет document replication.
