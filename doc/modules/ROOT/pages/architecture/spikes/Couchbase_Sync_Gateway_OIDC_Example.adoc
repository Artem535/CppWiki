= Couchbase Sync Gateway OIDC Example

== Goal

This spike captures the minimum Sync Gateway setup needed for the current CppWiki desktop sync path.

The current `401 Invalid login` symptom means Sync Gateway is accepting replication traffic, but it is not configured to trust the desktop OIDC token. In the current repo flow, the desktop client sends:

* `Authorization: Bearer <access_token>`
* `auth_mode = oidc_access_token_passthrough`
* workspace-scoped channels derived from trusted JWT claims

That only works when Sync Gateway validates the same OIDC issuer as the backend.

== Current claim model

The backend currently derives desktop sync channels from:

* `roles`
* `groups`

and always includes the user-scoped channel:

* `user:<sub>`

See:

* `src/server/service/sync_gateway_adapter.cc`
* `config/server.yaml`

== Recommended local-dev shape

Use this as a starting point for a local Sync Gateway deployment:

[source,json]
----
{
  "logging": {
    "console": {
      "enabled": true,
      "log_level": "info",
      "log_keys": ["*"]
    }
  },
  "api": {
    "public_interface": ":4984",
    "admin_interface": ":4985"
  },
  "bootstrap": {
    "server": "couchbase://couchbase",
    "username": "sync_gateway",
    "password": "sync_gateway_password",
    "use_tls": false
  },
  "oidc": {
    "default_provider": "authentik",
    "providers": {
      "authentik": {
        "issuer": "http://localhost:9000/application/o/cpp-wiki/",
        "client_id": "cppwiki-sync-gateway",
        "username_claim": "preferred_username",
        "groups_claim": "groups",
        "scope": ["openid", "profile", "email", "groups"]
      }
    }
  },
  "databases": {
    "cppwiki": {
      "bucket": "cppwiki",
      "enable_shared_bucket_access": true,
      "import_docs": true,
      "num_index_replicas": 0,
      "users": {
        "GUEST": {
          "disabled": true
        }
      },
      "sync": "function sync(doc, oldDoc) {\n  if (doc.channels && Array.isArray(doc.channels)) {\n    for (var i = 0; i < doc.channels.length; i++) {\n      channel(doc.channels[i]);\n    }\n    return;\n  }\n\n  if (doc.workspace_id) {\n    channel('workspace:' + doc.workspace_id);\n    return;\n  }\n\n  channel('public');\n}"
    }
  }
}
----

== Why this shape

. The OIDC provider must match the same issuer the backend validates.
. The desktop client already expects token passthrough, not local Sync Gateway users.
. The sync function should stop using a blanket `public` channel for real documents.
. Workspace-scoped access should come from document metadata, usually `doc.workspace_id` or an explicit `doc.channels` array.

== Important caveats

* Exact OIDC field names can vary a little across Sync Gateway versions. Verify the provider section against the installed SG version if the parser rejects it.
* If your Authentik app does not emit a `groups` claim, either add it there or change `groups_claim` to the actual claim name.
* `client_id` here is a placeholder. Replace it with the Sync Gateway OIDC client registration you created in Authentik.
* The sync function above is a starter. If your documents use a different workspace field, update the function to match the persisted schema.

== Local verification

After applying the config:

. Restart Sync Gateway.
. Confirm `curl http://127.0.0.1:4984/` returns `Welcome`.
. Request a desktop sync bootstrap from `cppwiki-server`.
. Start Couchbase Lite replication with a real Authentik access token.

If login still fails with `401 Invalid login`, check:

* issuer mismatch;
* wrong client registration;
* missing claims in the token;
* wrong `groups_claim` / `username_claim`;
* Sync Gateway database still using a local `GUEST` flow instead of OIDC.
