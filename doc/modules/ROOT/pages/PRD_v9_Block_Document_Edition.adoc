= Product Requirements Document (PRD)

*Wiki Platform v9 — Block Document Edition*

*Версия:* 9.0 +
*Дата:* 2026-06-11 +
*Статус:* Draft / Architecture Revision +
*Предыдущая версия:* PRD v8 — Confluence Sync Edition +
*Ключевое изменение:* переход от `Djot-first {plus} QTextEdit` к
`Block Document-first {plus} QWebEngine {plus} BlockNote/Tiptap`.

'''''

== 1. Executive Summary

Wiki Platform v9 — это desktop-first, offline-first wiki-платформа для
корпоративной базы знаний с архитектурой уровня Confluence / Outline /
Notion.

Главная архитектурная поправка v9: источником истины становится не
plain-text разметка Djot, а структурированный *Block Document Model*.

Приложение строится вокруг гибридной архитектуры:

[source,text]
----
Qt 6 Desktop Shell
    ↓
QWebEngine
    ↓
BlockNote / Tiptap / ProseMirror
    ↓
Document JSON
    ↓
C++ Core
    ↓
Couchbase Lite + Sync Gateway
    ↓
oat++ Backend
----

Расширяемость обеспечивается через безопасную систему плагинов:

[source,text]
----
C++ Plugin SDK
    ↓
WASM
    ↓
Wasmtime
----

В MVP используется упрощенная модель коллаборации:

[source,text]
----
Single Writer / Multi Reader
----

При этом архитектура сразу проектируется под обязательную будущую
realtime-коллаборацию через:

[source,text]
----
BlockNote
    ↓
Tiptap
    ↓
ProseMirror
    ↓
Yjs
    ↓
WebSocket Provider
----

'''''

== 2. Product Vision

Создать современную корпоративную wiki-платформу, которая сочетает:

[arabic]
. удобство Notion-like блочного редактора;
. надежность desktop-приложения;
. offline-first хранение;
. безопасную систему расширений;
. синхронизацию с Confluence;
. future-ready архитектуру для realtime collaboration;
. AI-friendly кодовую базу, удобную для разработки с Cursor/GitHub
Copilot.

Продукт должен позволять организациям постепенно заменить Confluence или
использовать его в гибридном режиме.

== 4. Delivery Discovery Alignment

=== 4.1. Problem Statement

Текущий стек v8 с `Djot-first {plus} QTextEdit` не дает достаточно
гибкую основу для Notion-like блочного редактора, structured document
storage, future realtime collaboration и безопасного расширения через
плагины. v9 должен перевести продукт на block document-first архитектуру
так, чтобы MVP оставался поставляемым без полного realtime collaboration
и без попытки сразу повторить весь Confluence/Notion surface.

=== 4.2. User Outcome

Пользователь должен иметь возможность создавать, читать, редактировать,
хранить и синхронизировать корпоративные wiki-документы в desktop-first
offline-first приложении без потери структуры документа и с понятной
моделью доступа.

=== 4.3. Critical Journeys and Failure-Sensitive Paths

[width="100%",cols="<25%,<25%,<25%,<25%",options="header",]
|===
|Journey / Path |Expected Outcome |Failure Mode |Required Handling
|Offline document edit |User edits and saves a block document locally
|Local storage write fails or document JSON becomes invalid |Block save,
preserve previous version, show recovery action

|Sync after offline work |Local changes replicate when network returns
|Conflict, auth failure, or rejected replication |Keep local copy, show
sync status, require explicit resolution path

|Confluence import/export |Supported page content maps to internal block
model |Unsupported macro or lossy conversion |Preserve source reference,
mark unsupported blocks, avoid silent data loss

|Plugin execution |Plugin extends document behavior safely |Plugin
crash, timeout, or unsafe operation |Sandbox via WASM, timeout
execution, disable plugin without crashing core

|Authenticated workspace access |User sees only permitted workspace
content |Token expired or access denied |Re-authenticate, fail closed,
do not expose cached unauthorized content
|===

=== 4.4. Success Criteria

[width="100%",cols="<25%,<25%,<25%,<25%",options="header",]
|===
|Category |Metric |Target |How Measured
|Product |Core wiki workflows completed in MVP
|Create/read/edit/search/sync available for supported documents |Manual
acceptance and QA scenarios

|Quality |Document round-trip integrity |No silent loss for supported
block types |Fixture-based import/save/export tests

|Latency |Local editor responsiveness |Editing remains interactive for
target document sizes |Desktop profiling and UX smoke tests

|Cost |Required infrastructure for MVP |Runs with self-hosted backend
and no mandatory paid AI service |Deployment review

|Safety |Plugin isolation |Plugin failure cannot crash or corrupt core
document state |Plugin sandbox tests

|Operations |Sync and auth observability |Sync/auth failures visible to
user and logs |Runbook and logging review
|===

=== 4.5. Deterministic vs ML/LLM Behavior

v9 MVP is primarily deterministic product and platform work. Core
requirements such as document storage, authorization, sync, rendering,
plugin isolation, import/export and conflict handling must not depend on
LLM output.

AI/LLM features are explicitly non-goals for MVP. If future AI
capabilities are added, they must be scoped through the ML/LLM delivery
pipeline with separate PRD-lite, Service Spec, Evaluation Plan, Security
Note and Prompt Template.

=== 4.6. Smallest Shippable Slice

The smallest shippable slice for v9 is:

* Qt desktop shell with QWebEngine-hosted block editor.
* Block document JSON as the source of truth.
* Local offline persistence.
* Single Writer / Multi Reader collaboration model.
* Basic sync status and Authentik-based authentication path.
* Minimal Confluence sync for supported content.
* WASM plugin sandbox proof for one non-critical extension point.

=== 4.7. Pilot / Release Gates

* Architecture gate: ADRs for editor choice, document model,
collaboration strategy, plugin system, rendering pipeline, Confluence
integration and auth model are approved.
* Quality gate: supported block fixtures pass round-trip tests and
invalid document states are rejected safely.
* Pilot gate: offline edit, sync recovery, auth renewal and plugin
failure scenarios pass acceptance testing.
* Release gate: runbook, security note and rollback path exist for
sync/auth/backend failures.

== 8. Major Architecture Changes from v8

=== 8.1. Removed from core architecture

* QTextEdit as main editor.
* QSyntaxHighlighter as main editing UX.
* Djot as primary storage format.
* Yjs ↔ QTextDocument bridge.
* Djot-first render pipeline.

=== 8.2. Added to core architecture

* QWebEngine as editor runtime host.
* BlockNote/Tiptap as editor.
* ProseMirror-compatible document model.
* Document JSON as source of truth.
* C{plus}{plus} canonical render engine.
* C{plus}{plus}/WASM plugin SDK.
* Authentik-based OIDC auth model.
* Document lock-based MVP collaboration.
* Yjs-ready future collaboration architecture.

== 9. Functional Requirements — UI

=== FR-UI-1. Desktop shell

The application shall use Qt 6 Widgets as the native desktop shell.

The shell shall include:

* main window;
* page tree;
* workspace selector;
* document tabs;
* sync status panel;
* account menu;
* settings dialog;
* plugin manager;
* Confluence sync status;
* offline/online indicator.

=== FR-UI-2. Editor host

The main editor shall run inside `QWebEngineView`.

The editor bundle shall be built as a local web asset and loaded from
application resources or local filesystem.

=== FR-UI-3. Editor framework

The editor shall use BlockNote as the primary editor framework.

BlockNote is selected because it provides:

* block-based editing;
* Notion-like UX;
* slash commands;
* custom blocks;
* structured document format;
* compatibility with Tiptap/ProseMirror ecosystem;
* path to Yjs collaboration.

=== FR-UI-4. QWebChannel bridge

The desktop shell shall expose a QWebChannel bridge to the editor
runtime.

The bridge shall expose APIs:

[source,text]
----
wiki.auth.*
wiki.documents.*
wiki.render.*
wiki.plugins.*
wiki.sync.*
wiki.confluence.*
wiki.presence.*
wiki.locks.*
----

=== FR-UI-5. Page tree

The page tree shall be implemented in Qt using Model/View architecture.

Required features:

* hierarchical pages;
* drag and drop reorder;
* workspace filtering;
* icons by page type;
* sync status badges;
* lock status badges.

=== FR-UI-6. Editor status

The editor shall display:

* current page title;
* saving status;
* lock owner;
* last saved time;
* local/offline status;
* sync status;
* Confluence link status;
* current user presence.

=== FR-UI-7. Read-only mode

When another user owns the document lock, the editor shall open in
read-only mode.

The user shall see:

* who is editing;
* when the lock was acquired;
* whether lock is active;
* option to request takeover if allowed by permissions.

'''''

== 10. Functional Requirements — Document Model

=== FR-DOC-1. Source of truth

The source of truth shall be a structured JSON document.

Djot, Markdown, HTML, ADF and Confluence Storage Format are
import/export formats, not canonical internal storage.

=== FR-DOC-2. Page document structure

A page document shall include:

[source,json]
----
{
  "id": "page-uuid",
  "type": "wiki_page",
  "workspace_id": "workspace-uuid",
  "title": "Architecture",
  "parent_id": "parent-page-uuid",
  "created_at": "2026-06-11T10:00:00Z",
  "updated_at": "2026-06-11T10:30:00Z",
  "created_by": "user-uuid",
  "updated_by": "user-uuid",
  "version": 17,
  "document": {
    "schema_version": "1.0",
    "blocks": []
  },
  "acl": {},
  "confluence": {},
  "sync": {}
}
----

=== FR-DOC-3. Block structure

Each block shall include:

[source,json]
----
{
  "id": "block-uuid",
  "type": "paragraph",
  "props": {},
  "content": [],
  "children": []
}
----

=== FR-DOC-4. Standard block types

MVP shall support:

* paragraph;
* heading;
* bullet list;
* numbered list;
* checklist;
* quote;
* code block;
* table;
* image;
* file attachment;
* divider;
* callout;
* link preview.

=== FR-DOC-5. Custom block types

Custom block types shall be registered by plugins.

Examples:

* mermaid;
* youtube;
* jira-issue;
* plantuml;
* drawio;
* confluence-macro-placeholder.

=== FR-DOC-6. Schema versioning

The document format shall include `schema++_++version`.

The application shall support migrations between document schema
versions.

=== FR-DOC-7. Stable block IDs

Every block shall have a stable UUID.

Stable block IDs are required for:

* comments;
* references;
* partial rendering;
* sync;
* future realtime collaboration;
* Confluence mapping.

'''''

== 11. Functional Requirements — Rendering

=== FR-REN-1. Two rendering layers

The product shall distinguish between:

[arabic]
. Editor Render.
. Canonical Render.

=== FR-REN-2. Editor Render

Editor Render is performed inside BlockNote.

It is responsible for:

* interactive editing;
* input controls;
* block handles;
* drag and drop;
* slash commands;
* inline previews.

Editor Render is not the source of truth for export output.

=== FR-REN-3. Canonical Render

Canonical Render is performed by the C{plus}{plus} RenderService.

It is responsible for:

* preview;
* read-only page view;
* HTML export;
* PDF export;
* Confluence export;
* search indexing;
* server-side rendering when needed.

Pipeline:

[source,text]
----
Document JSON
    ↓
Internal Document AST
    ↓
RenderService
    ↓
WASM plugin renderers
    ↓
HTML / ADF / Storage Format / Plain Text
----

=== FR-REN-4. Plugin render modes

Plugins shall support render modes:

[source,cpp]
----
enum class RenderMode {
    EditorPreview,
    ReadOnlyHtml,
    ExportHtml,
    ExportConfluenceStorage,
    ExportADF,
    SearchText
};
----

=== FR-REN-5. Sanitization

All HTML produced by plugins shall be sanitized before display in
QWebEngine or export.

=== FR-REN-6. Partial rendering

RenderService should support rendering a single block by ID.

This is required for:

* fast preview;
* plugin preview;
* block-level cache;
* future collaborative updates.

'''''

== 12. Functional Requirements — Plugin System

=== FR-PL-1. Runtime

Plugins shall run as WebAssembly modules in Wasmtime.

=== FR-PL-2. Plugin developer experience

Plugin authors shall write C{plus}{plus}20 code using the Wiki Plugin
SDK.

The SDK shall hide low-level ABI, memory transfer, serialization and
error handling.

=== FR-PL-3. Plugin manifest

Each plugin shall provide a manifest:

[source,json]
----
{
  "id": "mermaid",
  "name": "Mermaid Diagrams",
  "version": "1.0.0",
  "vendor": "Wiki Platform",
  "block_types": ["mermaid"],
  "permissions": [],
  "render_modes": [
    "EditorPreview",
    "ReadOnlyHtml",
    "ExportHtml",
    "SearchText"
  ]
}
----

=== FR-PL-4. Plugin schema

Each custom block shall expose a schema.

Example:

[source,json]
----
{
  "type": "mermaid",
  "title": "Mermaid Diagram",
  "fields": {
    "source": {
      "type": "multiline_text",
      "required": true
    },
    "theme": {
      "type": "select",
      "values": ["default", "dark", "forest"]
    }
  }
}
----

=== FR-PL-5. Editor integration

The editor shall not require a manually written JS tool for every
plugin.

Instead, the app shall include a generic custom block UI that can render
plugin-provided schemas.

For advanced blocks, plugins may ship optional frontend assets.

=== FR-PL-6. Plugin security

Plugins shall not have direct access to:

* host filesystem;
* process memory;
* network;
* system credentials;
* keyring;
* other plugins.

Any capability must be explicitly declared in the manifest and approved
by policy.

=== FR-PL-7. Plugin failures

A plugin crash, infinite loop or memory error shall not crash the host
application.

The host shall:

* terminate plugin execution;
* log the failure;
* mark plugin as unhealthy;
* show fallback block rendering.

=== FR-PL-8. Plugin SDK public API

The C{plus}{plus} SDK should expose an API similar to:

[source,cpp]
----
namespace wiki {

class Plugin {
public:
    virtual ~Plugin() = default;

    virtual PluginManifest manifest() const = 0;
    virtual BlockSchema schema(std::string_view block_type) const = 0;

    virtual ValidateResult validate(const Block& block) = 0;

    virtual RenderResult render(
        const Block& block,
        const RenderContext& context
    ) = 0;
};

}
----

The ABI glue shall be generated or provided by the SDK.

'''''

== 13. Functional Requirements — Collaboration

=== FR-COL-1. Strategic requirement

Realtime collaboration is a mandatory future feature.

The MVP may ship without full realtime editing, but architecture must
not block future realtime collaboration.

=== FR-COL-2. MVP model

MVP shall use:

[source,text]
----
Single Writer / Multi Reader
----

Only one user can actively edit a page at a time.

Other users may view the latest saved version and presence information.

=== FR-COL-3. Document lock

A document lock shall include:

[source,json]
----
{
  "page_id": "page-uuid",
  "owner_user_id": "user-uuid",
  "owner_session_id": "session-uuid",
  "acquired_at": "2026-06-11T10:00:00Z",
  "heartbeat_at": "2026-06-11T10:00:10Z",
  "expires_at": "2026-06-11T10:01:00Z",
  "mode": "exclusive_edit"
}
----

=== FR-COL-4. Lock heartbeat

The editor shall send lock heartbeat every 10 seconds.

If heartbeat is not received within configured timeout, the lock
expires.

=== FR-COL-5. Lock takeover

Users with sufficient permissions may force release a stale lock.

Force release shall be audited.

=== FR-COL-6. Presence

The application shall show users currently viewing or editing the
document.

Presence shall be delivered through WebSocket.

=== FR-COL-7. Phase 2 realtime architecture

Phase 2 shall introduce realtime editing using:

[source,text]
----
BlockNote
    ↓
Tiptap
    ↓
ProseMirror
    ↓
Yjs
    ↓
WebSocket Provider
    ↓
oat++ collaboration gateway or Hocuspocus-compatible service
----

=== FR-COL-8. Phase 2 features

Phase 2 shall support:

* simultaneous editing;
* user cursors;
* user selections;
* awareness;
* conflict-free updates;
* block-level collaboration;
* persistence of collaborative document state.

=== FR-COL-9. Phase 3 offline collaboration

Phase 3 shall support:

* concurrent offline edits;
* merge after reconnect;
* CRDT-based version history;
* conflict-free document convergence.

'''''

== 14. Functional Requirements — Storage

=== FR-DB-1. Local storage

Local data shall be stored in Couchbase Lite.

=== FR-DB-2. Stored entities

The local database shall store:

* users cache;
* workspaces;
* pages;
* documents;
* attachments metadata;
* sync metadata;
* lock cache;
* plugin metadata;
* Confluence mapping.

=== FR-DB-3. Attachments

Attachments shall be stored as local blobs with metadata in the
database.

Attachment references in documents shall use stable attachment IDs.

=== FR-DB-4. Sync metadata

Each page shall include sync metadata:

[source,json]
----
{
  "sync": {
    "local_revision": "local-rev",
    "remote_revision": "remote-rev",
    "last_sync_at": "2026-06-11T10:00:00Z",
    "status": "synced"
  }
}
----

=== FR-DB-5. Replication

Couchbase Lite shall replicate with Couchbase Sync Gateway when online.

=== FR-DB-6. Channels

Sync Gateway channels shall be used to restrict access to workspaces and
pages.

Channel examples:

[source,text]
----
workspace:{workspace_id}
page:{page_id}
team:{team_id}
----

'''''

== 15. Functional Requirements — Auth and Authorization

=== FR-AUTH-1. Identity provider

The recommended identity provider shall be Authentik.

Authentik shall be used as an OIDC/OAuth2 provider for:

* desktop app login;
* backend API authentication;
* Sync Gateway OIDC authentication;
* group and role claims.

=== FR-AUTH-2. Supported auth protocols

The platform shall support:

* OpenID Connect;
* OAuth2 Authorization Code Flow with PKCE;
* refresh tokens;
* JWT validation via JWKS.

=== FR-AUTH-3. Desktop login flow

The desktop app shall use system browser login.

Flow:

[source,text]
----
Desktop App
    ↓
Open system browser
    ↓
Authentik login
    ↓
Redirect to localhost callback or custom URI scheme
    ↓
Desktop receives authorization code
    ↓
Desktop exchanges code for tokens
    ↓
Tokens stored in system keyring
----

=== FR-AUTH-4. Token storage

Tokens shall be stored only in system keyring:

* Windows Credential Manager;
* macOS Keychain;
* Linux Secret Service.

Tokens shall not be stored in plain text config files.

=== FR-AUTH-5. Backend API auth

The oat{plus}{plus} backend shall validate access tokens.

Validation shall use:

* issuer validation;
* audience validation;
* signature validation through JWKS;
* expiration validation;
* required claims validation.

=== FR-AUTH-6. User identity mapping

The platform user ID shall be derived from stable OIDC `sub`.

User profile may include:

[source,json]
----

----

=== 21.3. Backend authorization flow

[source,text]
----
Request with Bearer token
    ↓
oat++ Auth Middleware
    ↓
Validate JWT with JWKS
    ↓
Extract sub/groups/roles
    ↓
Map to platform permissions
    ↓
Allow or deny
----

=== 21.4. Sync Gateway authorization flow

[source,text]
----
Desktop app obtains OIDC token
    ↓
Couchbase Lite replication starts
    ↓
Sync Gateway validates OIDC token
    ↓
roles_claim/groups_claim maps user roles
    ↓
Sync Gateway assigns channels
    ↓
Only permitted documents replicate
----

== 27. Appendix — Glossary

=== Block Document Model

Structured JSON document where every content unit is a typed block.

=== Editor Render

Interactive rendering inside the editor.

=== Canonical Render

Authoritative rendering for preview/export/search/sync.

=== Single Writer / Multi Reader

Collaboration model where only one user edits a document at a time.

=== OIDC

OpenID Connect authentication protocol.

=== Authentik

Self-hosted identity provider used for SSO and authorization claims.

=== Sync Gateway Channels

Couchbase Sync Gateway mechanism for document-level access filtering.

=== WASM Plugin

Sandboxed plugin compiled to WebAssembly.
