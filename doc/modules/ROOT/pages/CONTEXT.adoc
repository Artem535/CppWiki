= CppWiki — Context Glossary

Single bounded context: the CppWiki desktop app, its `cppwiki_server` backend, and their
shared sync/collaboration model. This glossary captures project-specific terms sharpened
during design discussions; general programming concepts are intentionally left out.

== Language

*Editability gate*:
The check in `EditorBridge` that decides whether a document mutation (rename, move,
delete, snapshot update) is allowed to proceed. It has two independent sources of "no":
another party holding the document's collaboration lock, and the document having an
unresolved sync conflict (see ADR-010, ADR-013). A mutation is rejected the same way
regardless of which source triggered it.
_Avoid_: "read-only mode" (too vague — always ask *why* it's read-only).

*Locked document*:
A document another collaborator currently holds the edit lock on, per the lock model in
`src/sync/`. Distinct from a *conflicted document* — both are read-only, but for different
reasons, and the document tree shows a different indicator for each.
_Avoid_: "blocked document", "read-only document" (conflates with conflicted state).

*Conflicted document*:
A document with an unresolved sync conflict between its local and remote versions,
tracked by `DocumentSyncService`/`DocumentSyncSnapshot`. Read-only until resolved (Use
local / Use remote / Merge) — see ADR-013.
_Avoid_: "locked document" (that's the other read-only source).

*Conflict resolution window*:
The standalone, non-modal window that shows the conflict list and local/remote comparison
for the currently open document. Auto-opens when the user opens or switches to a
conflicted document; does not live inside the sync-status panel/dialog and is not opened
by clicking the sync icon.
_Avoid_: "merge dialog" (legacy name from before this redesign — the underlying
`ConflictMergeDialog` widget is being repurposed into this window, but "merge dialog"
undersells that it now auto-opens and is conflict-scoped, not sync-panel-scoped).

*Sync panel* / *Sync details dialog*:
The dialog opened via the sync icon, showing sync/auth/repository status. After this
redesign it no longer contains the conflict list or resolution actions — only an
aggregate "N conflicts" indicator with no actions of its own.

*BlockNote AI (MVP scope)*:
For issue #3, exactly two AI capabilities: rewriting/improving a selected block via the
formatting toolbar, and autocomplete ("continue writing") via a slash-menu command. Both
share the same `@blocknote/xl-ai` `AIExtension` plumbing and the same custom transport
(see below). Full chat-agent/RAG workflows are explicitly out of scope for this MVP.
_Avoid_: "AI chat" (that's the separate, later rail mode described below — see ADR-014).

*Workspace rail*:
The narrow icon panel on the left edge of `MainWindow` for switching between full-view
modes: Documents, AI Chat, Code. Selecting a rail entry fully replaces the central
content area (document tree + `Page` are hidden entirely in AI Chat/Code modes, not kept
alongside them). See ADR-014.
_Avoid_: "tabs" (these are not `QTabWidget` tabs above the content; they are rail-driven
full-mode swaps).

*AI Chat mode*:
A rail mode presenting a list of chat sessions plus the active session's message thread.
Distinct from *BlockNote AI (MVP scope)* above: AI Chat is a standalone conversational
surface whose agent tools act on wiki documents (read/search/edit through existing
`EditorBridge`-gated methods), not an in-editor rewrite/autocomplete feature. See
ADR-014, ADR-015.

*Code mode*:
A rail mode built on the same agent engine as AI Chat, but with tools scoped to an
arbitrary user-opened local directory (not the wiki workspace): file read/write and
shell command execution, gated by a configurable permission mode (ask always / allow for
session / auto-accept). Implemented as native Qt widgets (no browser layer), with code
rendered via `KSyntaxHighlighting`. See ADR-015.
_Avoid_: "IDE" (Code mode is agent-driven/"vibecoding"-oriented, not a general-purpose
manual code editor).

*Agent engine*:
The shared C++ tool-use/function-calling loop (OpenAI-compatible chat-completions API)
underlying both AI Chat and Code mode. One engine, two tool registries — not two separate
agent implementations. Distinct from *AI chat transport* below, which is a simple text
streaming relay for the unrelated BlockNote AI MVP. See ADR-015.

*forkd*:
A microVM sandbox runtime (Linux/KVM-only) that forks sub-agent worker processes from a
warmed parent snapshot via copy-on-write, used for Code mode's sub-agent fan-out
isolation on Linux. Windows/macOS have no equivalent sandboxing in the MVP; the
permission-mode confirmations are the only safety mechanism there. See ADR-015.

*AI chat transport*:
The custom `ChatTransport` implementation (matching the `ai` SDK's interface consumed by
BlockNote's `AIExtension`) that relays AI requests through `EditorBridge`/`QWebChannel`
instead of `fetch`-ing a URL directly from browser JS. This is what keeps `QWebEngineView`
free of direct network access even for AI features — see ADR-012.
_Avoid_: "AI proxy" (the relay is bridge-based, not a network proxy).

*Accent color*:
A user-chosen theme color (one of four presets: blue/violet/orange/green, blue default)
applied as a near-transparent tint to the workspace rail's active-mode highlight and the
collaboration panel's `viewing` state. Distinct from the collaboration panel's other
three semantic status colors (editing/read-only/attention), which never change with the
accent preset. Configured via round swatch widgets on the Settings dialog's General
page. See ADR-016.
_Avoid_: "theme color" (this app doesn't have a light/dark theme choice — "accent color"
specifically means this one configurable highlight color).

*Document kind*:
The discriminator on a `Document` selecting which content schema/renderer applies: the
default wiki page (BlockNote JSON), a Jupyter notebook (nbformat JSON, view/edit only,
no execution), or an Excalidraw canvas (Excalidraw scene JSON, fully editable). All
kinds share the same sync/lock/conflict machinery (ADR-010/ADR-013); only the content
shape and renderer differ. See ADR-017.
_Avoid_: "document type" when referring to code (the field/discriminator is "kind" in
the design decision — keep terminology consistent once implemented).

*cppwiki-admin*:
The administrative tooling surface, delivered separately from the main `cppwiki` desktop
app (never as a mode inside it). Ships as a CLI first (C++, `ftxui`), with a web panel as
a distinct, later, not-yet-started piece. See `doc/backlog/Admin_Surface_Backlog.md`.
_Avoid_: "admin mode", "admin tab" (there is no admin surface inside the main app beyond
the temporary, soon-to-be-removed `New workspace` testing button).
